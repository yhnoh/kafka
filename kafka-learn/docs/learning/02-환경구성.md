# 02. Kafka 환경 구성 (Docker)

> 참고: "실전 카프카 개발부터 운영까지" — 2장

---

## 환경 구성

| 컴포넌트 | 이미지 | 수량 | 포트 |
|---|---|---|---|
| ZooKeeper | `confluentinc/cp-zookeeper:7.5.0` | 1 | 2181 |
| Kafka Broker | `confluentinc/cp-kafka:7.5.0` | 3 | 9092, 9093, 9094 |

### 왜 Confluent 이미지인가?

- "실전 카프카" 책에서 Confluent 이미지를 사용
- 환경변수 네이밍(`KAFKA_` 접두사)이 일관적이라 설정을 이해하기 쉬움
- Kafka Connect, Schema Registry 등 에코시스템 확장이 용이
- Kafka 코어 자체는 Apache Kafka와 동일

### 왜 브로커 3대인가?

- 리플리케이션 팩터(replication factor)를 체감하려면 최소 3대 필요
- 1대만 있으면 복제의 의미가 없음
- 실제 운영에서도 최소 3대가 권장사항

## ZooKeeper의 역할

Spring으로 비유하면 **Eureka(서비스 디스커버리) + Config Server**를 합친 것이다.

```
ZooKeeper가 관리하는 것:
├── 브로커 목록 — "현재 살아있는 브로커가 누구인가"
├── 토픽 메타데이터 — "이 토픽의 파티션이 어떤 브로커에 있는가"
├── 리더 선출 — "파티션의 리더 브로커가 죽으면 누가 대체하는가"
└── 컨트롤러 선출 — "브로커 중 관리자 역할은 누가 하는가"
```

> 나중에 KRaft 모드로 제거된 이유: Kafka가 스스로 이걸 할 수 있는데, 왜 외부 시스템에 의존하나?

---

## ZooKeeper vs KRaft

| | ZooKeeper 모드 | KRaft 모드 (Kafka 3.3+) |
|---|---|---|
| 구조 | Kafka + ZooKeeper 별도 프로세스 | Kafka만으로 동작 |
| 운영 복잡도 | ZooKeeper 관리 필요 | 단순 |
| 현업 상황 | 아직 많이 사용 중 | 신규 구축 시 권장 |
| 학습 가치 | ZooKeeper 역할을 이해할 수 있음 | 최신 아키텍처 이해 |

→ 학습 목적으로 ZooKeeper 모드로 시작, 나중에 KRaft 마이그레이션도 학습 예정

## docker-compose.yml 핵심 설정 해설

### LISTENERS vs ADVERTISED_LISTENERS

Docker 환경에서 **가장 많이 부딪히는 문제 1위**이다.

```
KAFKA_LISTENERS: INTERNAL://0.0.0.0:29092,EXTERNAL://0.0.0.0:9092
  → 브로커가 실제로 바인딩하는 주소 (서버 소켓)

KAFKA_ADVERTISED_LISTENERS: INTERNAL://kafka-1:29092,EXTERNAL://localhost:9092
  → 클라이언트에게 "이 주소로 연결해" 라고 알려주는 주소
```

**왜 2개로 분리하는가?**

```
INTERNAL (kafka-1:29092)
  → 브로커 간 통신 (Docker 내부 네트워크)
  → 컨테이너 이름(kafka-1)으로 접근

EXTERNAL (localhost:9092)
  → 호스트에서 Spring 앱이 접근
  → localhost + 포트 매핑으로 접근

KAFKA_INTER_BROKER_LISTENER_NAME: INTERNAL
  → "브로커끼리는 INTERNAL 리스너를 사용해"
```

**흔한 실수: ADVERTISED_LISTENERS를 localhost로만 설정**

```
❌ 실패하는 설정:
  KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
  → 컨테이너 내부에서 localhost = 자기 자신
  → 브로커 간 통신 불가 → 토픽 생성 timeout

✅ 올바른 설정:
  리스너를 INTERNAL/EXTERNAL로 분리
  INTERNAL은 Docker 네트워크 주소, EXTERNAL은 호스트 주소
```

### 리플리케이션 관련 설정

```yaml
KAFKA_DEFAULT_REPLICATION_FACTOR: 3
  # 토픽 데이터를 3개 브로커에 모두 복제
  # 브로커 1대가 죽어도 데이터 유실 없음

KAFKA_MIN_INSYNC_REPLICAS: 2
  # "최소 2개 브로커에 쓰기 완료되어야 성공 응답"
  # acks=all과 함께 쓰면 데이터 유실 방지의 핵심

KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 3
  # __consumer_offsets 내부 토픽의 복제 팩터
  # 컨슈머 오프셋 정보도 안전하게 복제
```

## 토픽 생성과 구조 확인

### 토픽 생성

```bash
docker exec kafka-1 kafka-topics --create \
  --bootstrap-server kafka-1:29092 \
  --topic test-topic \
  --partitions 3 \
  --replication-factor 3
```

### 토픽 상태 확인

```bash
docker exec kafka-1 kafka-topics --describe \
  --bootstrap-server kafka-1:29092 \
  --topic test-topic
```

출력 예시:
```
Topic: test-topic  PartitionCount: 3  ReplicationFactor: 3  Configs: min.insync.replicas=2
  Partition: 0  Leader: 1  Replicas: 1,2,3  Isr: 1,2,3
  Partition: 1  Leader: 2  Replicas: 2,3,1  Isr: 2,3,1
  Partition: 2  Leader: 3  Replicas: 3,1,2  Isr: 3,1,2
```

```
Partition: 0  Leader: 1  Replicas: 1,2,3  Isr: 1,2,3
             ─────────  ──────────────── ────────────
             │           │                │
             │           │                └─ ISR (In-Sync Replicas)
             │           │                   리더와 동기화된 브로커 목록
             │           │
             │           └─ 이 파티션의 데이터가 존재하는 브로커
             │
             └─ 읽기/쓰기를 담당하는 브로커
```

- 파티션 0 → 브로커 1이 리더
- 파티션 1 → 브로커 2가 리더
- 파티션 2 → 브로커 3이 리더
- Kafka가 자동으로 리더를 분산 → 로드밸런싱

## 메시지 전송/소비 테스트 (CLI)

### 프로듀서 (메시지 전송)

```bash
docker exec kafka-1 kafka-console-producer \
  --bootstrap-server kafka-1:29092 \
  --topic test-topic
```

### 컨슈머 (메시지 소비)

```bash
docker exec kafka-1 kafka-console-consumer \
  --bootstrap-server kafka-1:29092 \
  --topic test-topic \
  --from-beginning
```

### Spring 앱에서 연결할 때

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092,localhost:9093,localhost:9094
```

→ EXTERNAL 리스너(localhost:9092/9093/9094)를 사용한다.

## ZooKeeper에서 브로커 등록 확인

```bash
docker exec zookeeper bash -c "echo 'ls /brokers/ids' | zookeeper-shell localhost:2181"
# 출력: [1, 2, 3]  ← 브로커 3대 모두 등록됨
```
