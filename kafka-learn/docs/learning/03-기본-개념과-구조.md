> 참고: "실전 카프카 개발부터 운영까지" — 3장

## 토픽 → 파티션 → 세그먼트 계층 구조

```
토픽 (Topic)
├── 파티션 0 (Partition)
│   ├── 세그먼트 0 (00000000000000000000.log)   ← offset 0~
│   ├── 세그먼트 1 (00000000000000001000.log)   ← offset 1000~
│   └── ...
├── 파티션 1
│   └── ...
└── 파티션 2
    └── ...
```

### RDB와의 비유

| Kafka | RDB |
|---|---|
| 토픽(Topic) | 테이블(Table) |
| 파티션(Partition) | 샤딩(Sharding)된 테이블 조각 |
| 세그먼트(Segment) | 테이블스페이스의 데이터 파일 |
| 메시지(Record) | 행(Row) |
| 오프셋(Offset) | Auto Increment PK |

차이점: DB는 랜덤 접근(`WHERE id=3`)이 가능하지만, Kafka는 순차 읽기 기반이다.

### 왜 이런 계층 구조인가?

**파티션이 없으면:**
- 하나의 토픽 = 하나의 큐
- 컨슈머 1대만 읽을 수 있음 (병렬 처리 불가)
- 초당 10만 건 → 컨슈머 1대가 다 감당해야 함

**세그먼트가 없으면:**
- 10GB짜리 파일에서 7일 지난 데이터만 삭제? → 파일 전체를 다시 써야 함
- 세그먼트로 쪼개면 → 오래된 세그먼트만 통째로 삭제 (빠르고 깔끔)

## 파티션의 실제 파일 구조

브로커 내부 경로: `/var/lib/kafka/data/{토픽명}-{파티션번호}/`

```
test-topic-0/
├── 00000000000000000000.log           ← 세그먼트 파일 (실제 메시지 데이터)
│                                         파일명 = 이 세그먼트의 시작 offset
├── 00000000000000000000.index         ← 오프셋 인덱스
│                                         "offset N은 .log 파일의 바이트 위치 X에 있다"
├── 00000000000000000000.timeindex     ← 타임스탬프 인덱스
│                                         "이 시간의 메시지는 offset N부터다"
│                                         → offset 리셋을 시간 기준으로 할 때 사용
├── leader-epoch-checkpoint            ← 리더 변경 이력
└── partition.metadata                 ← 파티션 메타정보
```

### 세그먼트 파일 내부 확인 방법

```bash
docker exec kafka-1 kafka-dump-log \
  --files /var/lib/kafka/data/test-topic-0/00000000000000000000.log \
  --print-data-log
```

출력 예시:
```
| offset: 0 CreateTime: 1774363548701 keySize: -1 valueSize: 11 sequence: 0 payload: hello-kafka
| offset: 1 CreateTime: 1774363548708 keySize: -1 valueSize: 14 sequence: 1 payload: partition-test
| offset: 2 CreateTime: 1774363548708 keySize: -1 valueSize: 16 sequence: 2 payload: replication-test
```

- `offset`: 파티션 내 순번
- `keySize: -1`: key 없음. key를 넣지 않으면 -1
- `payload`: 실제 메시지 내용

> key가 없으면 Sticky Partitioner가 동작하여 같은 배치의 메시지를 한 파티션에 모아 보낸다.

## 리플리케이션 (Replication)

### 핵심 개념

```
파티션 0:
  브로커 1: [msg0] [msg1] [msg2]  ← Leader (읽기/쓰기 담당)
  브로커 2: [msg0] [msg1] [msg2]  ← Follower (복제만)
  브로커 3: [msg0] [msg1] [msg2]  ← Follower (복제만)
```

- **Leader**: 프로듀서/컨슈머가 통신하는 대상. 읽기/쓰기 모두 리더를 통해서만 가능
- **Follower**: 리더의 데이터를 복제만 함. 리더가 죽으면 승격
- **ISR (In-Sync Replicas)**: 리더와 동기화 상태인 복제본 목록

### 리플리케이션 팩터별 안전성

| 설정 | 의미 | 장애 허용 |
|---|---|---|
| `replication-factor=1` | 복제 없음 | 브로커 1대 죽으면 데이터 유실 |
| `replication-factor=2` | 1개 복제본 | 1대 죽어도 안전 |
| `replication-factor=3` | 2개 복제본 | 2대 죽어도 안전 (권장) |

## 브로커 장애 실험 결과

### 실험 1: 브로커 1대 중단 (kafka-2 stop)

```
          죽기 전                    →  죽은 후
───────────────────────────────────────────────────
파티션 0  Leader: 1  ISR: 1,2,3     →  Leader: 1  ISR: 1,3
파티션 1  Leader: 2  ISR: 2,3,1     →  Leader: 3  ISR: 3,1  ← 리더 자동 변경!
파티션 2  Leader: 3  ISR: 3,1,2     →  Leader: 3  ISR: 3,1
```

- ISR에서 죽은 브로커(2)가 빠짐
- 파티션 1의 리더가 죽었으므로 → ISR 중 브로커 3이 자동 승격
- `acks=all` 메시지 전송: **성공** (ISR 2대 >= min.insync.replicas 2)

### 실험 2: 브로커 2대 중단 (kafka-2, kafka-3 stop)

```
파티션 0  Leader: 1  ISR: 1  (1대만 살아있음)
파티션 1  Leader: 1  ISR: 1
파티션 2  Leader: 1  ISR: 1
```

- `acks=all` 메시지 전송: **실패!**
- 에러: `NotEnoughReplicasException`
- 이유: ISR 1대 < min.insync.replicas 2 → 조건 불충족

### 실험 3: 브로커 복구 후

```
파티션 0  Leader: 1  ISR: 1,3,2  ← ISR 복구됨
파티션 1  Leader: 1  ISR: 1,2,3  ← ISR 복구됨
파티션 2  Leader: 1  ISR: 1,2,3  ← ISR 복구됨
```

- ISR은 복구되지만, **리더가 브로커 1에 몰려있음**
- 장애 전에는 리더가 분산(1,2,3)이었으나, 복구 후에도 자동으로 돌아가지 않음
- → **Preferred Replica Election** (7장 운영에서 다룸)

## acks 설정과 데이터 안전성

프로듀서가 "메시지를 보냈다"의 기준을 결정하는 설정.

| acks | 동작 | 데이터 안전성 | 성능 |
|---|---|---|---|
| `acks=0` | 전송만 하고 응답 안 기다림 | 유실 가능 | 가장 빠름 |
| `acks=1` | 리더에 쓰기 완료 시 응답 | 리더 죽으면 유실 가능 | 중간 |
| `acks=all(-1)` | ISR 전체에 복제 완료 시 응답 | 가장 안전 | 가장 느림 |

### acks=all + min.insync.replicas 조합이 핵심

```
replication-factor=3, min.insync.replicas=2, acks=all

→ "최소 2개 브로커에 복제 완료되어야 성공"
→ 1대 장애: 쓰기 가능 (ISR 2대)
→ 2대 장애: 쓰기 불가 (ISR 1대 < min.insync.replicas 2)
→ 데이터 유실: 불가능 (2곳에 저장되었으니)
```

### 트레이드오프

- `min.insync.replicas`가 높을수록 → 데이터 안전 / 가용성 감소
- `min.insync.replicas`가 낮을수록 → 가용성 증가 / 유실 위험
- 실무 권장: `replication-factor=3`, `min.insync.replicas=2`, `acks=all`
