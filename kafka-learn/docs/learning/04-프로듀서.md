# 04. 프로듀서 (Producer)

> 참고: "실전 카프카 개발부터 운영까지" — 4장

---

## 프로듀서 내부 동작 흐름

```
send() 호출
  │
  ▼
① Serializer ─── 객체를 바이트 배열로 변환 (key, value 각각)
  │
  ▼
② Partitioner ── 어떤 파티션으로 보낼지 결정
  │                 key 있음 → hash(key) % 파티션수
  │                 key 없음 → Sticky Partitioner (같은 배치는 한 파티션)
  │
  ▼
③ RecordAccumulator ── 파티션별 배치 버퍼에 메시지 쌓기
  │                      batch.size만큼 차거나
  │                      linger.ms 시간이 지나면
  │
  ▼
④ Sender (별도 스레드) ── 배치를 브로커로 전송
  │                        max.in.flight.requests.per.connection만큼 동시 전송
  │
  ▼
⑤ 브로커 응답 ── acks 설정에 따라 성공/실패 판단
                   실패 시 retries만큼 재시도
```

### Spring 비유

```
RestTemplate:  요청 1건 → 전송 → 응답 대기 → 다음 요청 (동기, 건건이)
KafkaProducer: 요청 N건 → 배치로 모음 → 한번에 전송 → 응답 (비동기, 묶어서)
```

`send()`를 호출하면 메시지가 바로 브로커에 가는 게 아니다. RecordAccumulator 버퍼에 쌓였다가 Sender 스레드가 배치 단위로 전송한다.

## 핵심 설정 — 카테고리별 정리

### 카테고리 1: 신뢰성 (데이터 유실 방지)

| 설정 | 기본값 | 설명 |
|---|---|---|
| `acks` | `all` (Kafka 3.0+) | 몇 개 브로커에 쓰기 완료되어야 성공으로 볼 것인가 |
| `retries` | `2147483647` | 전송 실패 시 재시도 횟수 |
| `delivery.timeout.ms` | `120000` (2분) | 재시도 포함 전체 전송 제한 시간 |
| `enable.idempotence` | `true` (Kafka 3.0+) | 중복 전송 방지 |
| `max.in.flight.requests.per.connection` | `5` | 응답 안 받고 동시에 보낼 수 있는 요청 수 |

### 카테고리 2: 성능 (처리량 최적화)

| 설정 | 기본값 | 설명 |
|---|---|---|
| `batch.size` | `16384` (16KB) | 배치 하나의 최대 크기 |
| `linger.ms` | `0` | 배치가 안 찼어도 이 시간 후 전송 |
| `buffer.memory` | `33554432` (32MB) | 프로듀서 전체 버퍼 크기 |
| `compression.type` | `none` | 압축 방식 (none, gzip, snappy, lz4, zstd) |

### 카테고리 3: 직렬화

| 설정 | 설명 |
|---|---|
| `key.serializer` | key를 바이트로 변환하는 방법 |
| `value.serializer` | value를 바이트로 변환하는 방법 |

## 각 설정 상세 설명

### acks — "메시지를 보냈다"의 기준

```
acks=0:
  Producer → 전송 → 끝 (응답 안 기다림)
  → 유실 가능. 로그/메트릭처럼 유실 허용 가능한 데이터에만 사용

acks=1:
  Producer → 전송 → 리더에 쓰기 완료 → 성공 응답
  → 리더가 응답 후 팔로워에 복제되기 전에 죽으면 유실

acks=all (-1):
  Producer → 전송 → 리더 + ISR 전체에 복제 완료 → 성공 응답
  → 가장 안전. min.insync.replicas와 함께 사용해야 의미 있음
```

#### acks=all과 min.insync.replicas의 관계

```
replication-factor=3, min.insync.replicas=2, acks=all

시나리오 1: 브로커 3대 모두 정상
  → ISR: 1,2,3 (3대) >= min.insync.replicas (2) → 전송 성공

시나리오 2: 브로커 1대 장애
  → ISR: 1,3 (2대) >= min.insync.replicas (2) → 전송 성공

시나리오 3: 브로커 2대 장애
  → ISR: 1 (1대) < min.insync.replicas (2) → NotEnoughReplicasException
```

### batch.size와 linger.ms — 처리량의 핵심

```
batch.size = 16KB (기본값)
linger.ms = 0 (기본값)

기본값일 때:
  메시지 도착 → 배치에 넣기 → linger.ms=0이니까 바로 전송
  → 사실상 배치를 안 쓰는 것과 같음
  → 메시지 1건마다 네트워크 왕복 → 처리량 낮음

linger.ms=20으로 변경하면:
  메시지 도착 → 배치에 넣기 → 20ms 대기 → 그 사이 들어온 메시지와 묶어서 전송
  → 네트워크 왕복 횟수 감소 → 처리량 증가
  → 대신 개별 메시지의 지연이 최대 20ms 증가
```

**트레이드오프:**
- `linger.ms`를 늘릴수록 → 처리량 증가 / 지연 증가
- `batch.size`를 늘릴수록 → 배치에 더 많은 메시지 → 처리량 증가 / 메모리 사용 증가

### enable.idempotence — 중복 전송 방지

```
멱등성이 없을 때:
  Producer → 전송 → 브로커 저장 완료 → 응답 전에 네트워크 끊김
  → Producer는 "실패했다"고 판단 → 재전송
  → 브로커에 같은 메시지가 2번 저장됨 (중복!)

멱등성이 있을 때:
  각 메시지에 (ProducerId + Sequence Number)가 붙음
  → 브로커가 "이미 받은 메시지"를 인식 → 중복 저장 방지
```

Kafka 3.0부터 기본값이 `true`이므로 별도 설정 불필요. 단, 아래 조건이 함께 충족되어야 함:
- `acks=all`
- `max.in.flight.requests.per.connection <= 5`

### max.in.flight.requests.per.connection — 순서 보장과의 관계

```
max.in.flight = 1:
  배치1 전송 → 응답 대기 → 배치2 전송 → 응답 대기
  → 순서 보장 확실 / 처리량 매우 낮음

max.in.flight = 5 (기본값):
  배치1,2,3,4,5 동시 전송 → 응답 대기
  → 처리량 높음

  ⚠️ 멱등성이 꺼져 있으면:
    배치1 실패 → 배치2 성공 → 배치1 재전송 성공
    → 순서가 뒤집힘! (2 → 1)

  ✅ 멱등성이 켜져 있으면:
    브로커가 Sequence Number로 순서 확인
    → 5 이하에서는 순서 보장됨
```

### compression.type — 압축

```
none:   압축 안 함. CPU 절약, 네트워크 비용 증가
gzip:   압축률 높음. CPU 많이 사용. 네트워크 대역폭 부족할 때
snappy: 중간 압축률. 빠른 속도. 범용적 선택
lz4:    낮은 압축률. 가장 빠른 속도. 실시간 처리에 적합
zstd:   높은 압축률. gzip보다 빠름. Kafka 2.1+
```

**실무 선택 기준:**
- 네트워크가 병목 → `gzip` 또는 `zstd`
- CPU가 병목 → `lz4` 또는 `snappy`
- 잘 모르겠으면 → `lz4` (가장 무난)

### buffer.memory — 전체 버퍼 크기

```
buffer.memory = 32MB (기본값)

RecordAccumulator가 사용하는 총 메모리 크기.
버퍼가 가득 차면 send()가 블로킹됨 (max.block.ms만큼 대기 후 예외 발생)

버퍼가 가득 차는 상황:
  - 브로커가 느려서 전송이 밀림
  - 프로듀서가 너무 빠르게 send() 호출
  → TimeoutException 또는 BufferExhaustedException
```

### delivery.timeout.ms — 전체 제한 시간

```
send() 호출부터 성공/실패 확정까지의 최대 시간.
이 안에 retries가 포함됨.

delivery.timeout.ms >= linger.ms + request.timeout.ms

기본값: 120000ms (2분)
→ 2분 안에 재시도까지 포함해서 결과가 나와야 함
→ 초과 시 TimeoutException
```

## Partitioner — 파티션 배정 전략

### key 기반 파티셔닝

```java
// key = orderId → 같은 주문의 이벤트는 항상 같은 파티션 → 순서 보장
kafkaTemplate.send("order-events", orderId.toString(), orderEventJson);
```

내부 동작: `hash(key) % 파티션수`
- 같은 key → 항상 같은 파티션 → 파티션 내 순서 보장
- **주의: 파티션 수를 변경하면 해시 결과가 달라짐 → 같은 key가 다른 파티션으로 갈 수 있음**

### key 없을 때 — Sticky Partitioner (Kafka 2.4+)

```
이전 방식 (Round-Robin):
  msg1 → 파티션 0, msg2 → 파티션 1, msg3 → 파티션 2
  → 배치가 잘 안 모임 → 네트워크 왕복 증가

Sticky Partitioner:
  msg1, msg2, msg3 → 파티션 0 (배치가 찰 때까지)
  msg4, msg5, msg6 → 파티션 1 (다음 배치)
  → 배치가 잘 모임 → 처리량 향상
```

### 커스텀 파티셔너

특수한 경우 직접 구현 가능:

```java
public class CustomPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        // VIP 고객은 전용 파티션으로
        if (key != null && key.toString().startsWith("VIP-")) {
            return 0;
        }
        // 나머지는 기본 해시
        return Utils.toPositive(Utils.murmur2(keyBytes)) % cluster.partitionCountForTopic(topic);
    }
}
```

## 전송 방식 3가지

### 1. Fire and Forget (발사 후 망각)

```java
kafkaTemplate.send("topic", "message");
// 결과를 확인하지 않음. 유실 가능성 있음.
// 로그, 메트릭 등 유실 허용 가능한 데이터에 사용
```

### 2. Synchronous (동기 전송)

```java
try {
    SendResult<String, String> result = kafkaTemplate.send("topic", "message").get();
    // .get()으로 블로킹 → 성공할 때까지 대기
    // 결과에서 파티션, 오프셋 확인 가능
    log.info("partition={}, offset={}",
        result.getRecordMetadata().partition(),
        result.getRecordMetadata().offset());
} catch (ExecutionException e) {
    // 전송 실패 처리
    log.error("전송 실패", e.getCause());
}
```

- 가장 안전하지만 가장 느림
- 매 전송마다 브로커 응답을 기다리므로 처리량이 낮음
- 정말 중요한 메시지(결제, 주문)에 사용

### 3. Asynchronous (비동기 전송 + 콜백)

```java
kafkaTemplate.send("topic", key, value)
    .whenComplete((result, ex) -> {
        if (ex != null) {
            log.error("전송 실패: topic={}, key={}", "topic", key, ex);
            // 실패 처리: 재시도, DLQ 저장, 알림 등
        } else {
            log.info("전송 성공: partition={}, offset={}",
                result.getRecordMetadata().partition(),
                result.getRecordMetadata().offset());
        }
    });
// send()는 즉시 리턴. 콜백은 나중에 실행됨.
```

- 실무에서 가장 많이 사용하는 방식
- 비동기라 처리량 높음 + 콜백으로 실패 처리 가능

## 실무에서 반드시 챙겨야 할 것들

### 1. key 설계는 신중하게

```
❌ 잘못된 key 설계:
  key = userId → 헤비 유저가 있으면 특정 파티션에 메시지 몰림 (Hot Partition)

✅ 좋은 key 설계:
  key = orderId → 주문 단위로 순서 보장이 필요한 경우
  key = null → 순서가 중요하지 않으면 key 없이 보내서 균등 분배
```

**규칙: "순서 보장이 필요한 단위"를 key로 쓴다.**

### 2. 파티션 수 변경하면 key 라우팅이 깨진다

```
파티션 3개일 때: hash("order-123") % 3 = 1  → 파티션 1
파티션 5개로 늘리면: hash("order-123") % 5 = 3  → 파티션 3 (!!)

→ 같은 key인데 다른 파티션으로 감
→ 순서 보장이 깨짐
→ 파티션 수는 처음에 넉넉하게 설정하는 게 중요
```

### 3. Callback에서 블로킹 하지 마라

```java
// ❌ 위험: 콜백 안에서 동기 호출
kafkaTemplate.send("topic", value).whenComplete((result, ex) -> {
    // Sender 스레드에서 실행됨!
    // 여기서 DB 호출 같은 블로킹 작업을 하면 Sender 스레드가 멈춤
    // → 다른 모든 전송도 멈춤 → 처리량 급감
    repository.save(something);  // ❌
});

// ✅ 안전: 콜백에서는 가벼운 작업만, 무거운 작업은 별도 스레드로
kafkaTemplate.send("topic", value).whenComplete((result, ex) -> {
    if (ex != null) {
        log.error("실패", ex);
        // 별도 스레드나 큐로 실패 처리 위임
        failureQueue.add(new FailedRecord(value, ex));
    }
});
```

### 4. 직렬화 실패는 재시도해도 소용없다

```
Serialization 실패 = 메시지 자체가 잘못됨
→ retries를 아무리 해도 같은 에러
→ SerializationException은 별도로 잡아서 DLQ(Dead Letter Queue)로 보내야 함
```

### 5. 실무 권장 설정 조합

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092,localhost:9093,localhost:9094
    producer:
      # 신뢰성
      acks: all
      retries: 2147483647        # 기본값. delivery.timeout.ms 내에서 계속 재시도
      # 성능
      batch-size: 32768          # 32KB (기본 16KB보다 약간 큰 값)
      properties:
        linger.ms: 20            # 20ms 대기하며 배치 모으기
        compression.type: lz4    # 네트워크 절약
        # 멱등성 (Kafka 3.0+ 기본값이지만 명시적으로)
        enable.idempotence: true
        max.in.flight.requests.per.connection: 5
        # 제한 시간
        delivery.timeout.ms: 120000
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.apache.kafka.common.serialization.StringSerializer
```

### 6. 프로듀서 모니터링 — 반드시 봐야 할 메트릭

| 메트릭 | 의미 | 경고 기준 |
|---|---|---|
| `record-send-rate` | 초당 전송 메시지 수 | 급격한 변동 |
| `record-error-rate` | 초당 전송 실패 수 | 0이 아니면 즉시 확인 |
| `request-latency-avg` | 평균 전송 지연 | 비정상적 증가 |
| `batch-size-avg` | 평균 배치 크기 | 너무 작으면 linger.ms 조정 |
| `buffer-available-bytes` | 남은 버퍼 크기 | 0에 가까우면 위험 |

## Spring Kafka 프로듀서 실습 코드

### 1. 기본 프로듀서

```java
@Component
@RequiredArgsConstructor
public class SimpleProducer {

    private final KafkaTemplate<String, String> kafkaTemplate;

    /**
     * Fire and Forget 방식
     * 유실 허용 가능한 데이터 (로그, 메트릭)에 사용
     */
    public void sendFireAndForget(String topic, String message) {
        kafkaTemplate.send(topic, message);
    }

    /**
     * 비동기 + 콜백 방식 (실무 권장)
     */
    public void sendAsync(String topic, String key, String value) {
        kafkaTemplate.send(topic, key, value)
            .whenComplete((result, ex) -> {
                if (ex != null) {
                    log.error("전송 실패: topic={}, key={}, error={}", topic, key, ex.getMessage());
                } else {
                    RecordMetadata metadata = result.getRecordMetadata();
                    log.info("전송 성공: topic={}, partition={}, offset={}",
                        metadata.topic(), metadata.partition(), metadata.offset());
                }
            });
    }

    /**
     * 동기 방식
     * 반드시 전송 성공을 확인해야 하는 경우 (결제, 주문)
     */
    public void sendSync(String topic, String key, String value) {
        try {
            SendResult<String, String> result = kafkaTemplate.send(topic, key, value).get();
            RecordMetadata metadata = result.getRecordMetadata();
            log.info("전송 성공: partition={}, offset={}", metadata.partition(), metadata.offset());
        } catch (ExecutionException e) {
            log.error("전송 실패", e.getCause());
            throw new RuntimeException("Kafka 전송 실패", e.getCause());
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new RuntimeException("Kafka 전송 중 인터럽트", e);
        }
    }
}
```

### 2. key 기반 파티셔닝 확인 실습

```java
@Component
@RequiredArgsConstructor
public class PartitioningTest {

    private final KafkaTemplate<String, String> kafkaTemplate;

    /**
     * 같은 key는 항상 같은 파티션으로 가는지 확인
     */
    public void testKeyPartitioning() {
        String topic = "test-topic";

        // 같은 key("order-1")로 3번 전송 → 모두 같은 파티션이어야 함
        for (int i = 0; i < 3; i++) {
            kafkaTemplate.send(topic, "order-1", "message-" + i)
                .whenComplete((result, ex) -> {
                    log.info("key=order-1 → partition={}",
                        result.getRecordMetadata().partition());
                });
        }

        // 다른 key로 전송 → 다른 파티션으로 분배될 수 있음
        for (int i = 0; i < 3; i++) {
            kafkaTemplate.send(topic, "order-" + i, "message-" + i)
                .whenComplete((result, ex) -> {
                    log.info("key=order-{} → partition={}",
                        i, result.getRecordMetadata().partition());
                });
        }
    }
}
```

### 3. batch.size / linger.ms 차이 체감 실습

```yaml
# application-batch-test.yml
# 프로파일로 분리하여 비교 테스트 가능

# 프로파일 1: 배치 없음 (기본값)
spring.kafka.producer.properties.linger.ms: 0
spring.kafka.producer.batch-size: 16384

# 프로파일 2: 배치 적극 활용
spring.kafka.producer.properties.linger.ms: 50
spring.kafka.producer.batch-size: 65536
```

```java
@Component
@RequiredArgsConstructor
public class BatchPerformanceTest {

    private final KafkaTemplate<String, String> kafkaTemplate;

    /**
     * 1만 건 전송 시간 비교
     * linger.ms=0 vs linger.ms=50 으로 프로파일 바꿔가며 테스트
     */
    public void testBatchPerformance() {
        String topic = "test-topic";
        int messageCount = 10000;

        long start = System.currentTimeMillis();

        CountDownLatch latch = new CountDownLatch(messageCount);
        for (int i = 0; i < messageCount; i++) {
            kafkaTemplate.send(topic, "key-" + (i % 100), "payload-" + i)
                .whenComplete((result, ex) -> latch.countDown());
        }

        try {
            latch.await(30, TimeUnit.SECONDS);
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }

        long elapsed = System.currentTimeMillis() - start;
        log.info("{}건 전송 완료: {}ms (처리량: {} msg/sec)",
            messageCount, elapsed, messageCount * 1000L / elapsed);
    }
}
```

## 프로듀서 관련 장애 시나리오

### 시나리오 1: 브로커 전체 장애

```
증상: send() 후 delivery.timeout.ms(2분) 후에 TimeoutException
원인: 모든 브로커가 죽었거나 네트워크 단절
대응: buffer.memory가 가득 차면 send() 자체가 블로킹 → 앱 전체가 멈출 수 있음
→ max.block.ms 설정으로 블로킹 시간 제한 필요
```

### 시나리오 2: 특정 파티션의 리더 변경 중

```
증상: 일시적으로 NOT_LEADER_OR_FOLLOWER 에러
원인: 리더 브로커 교체 중 (롤링 배포 등)
대응: retries가 기본값(매우 큼)이므로 자동 재시도 → 대부분 자동 복구
```

### 시나리오 3: 메시지 크기 초과

```
증상: RecordTooLargeException
원인: 메시지 크기 > max.request.size (기본 1MB)
대응:
  - 큰 데이터는 외부 저장소(S3 등)에 넣고 참조만 Kafka로
  - 정말 필요하면 max.request.size와 브로커의 message.max.bytes 함께 조정
```
