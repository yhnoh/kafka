# 05. 컨슈머 (Consumer)

> 참고: "실전 카프카 개발부터 운영까지" — 5장, 6장

---

## 컨슈머 내부 동작 흐름

```
poll() 호출
  │
  ▼
① Fetcher ─── 브로커에서 메시지 가져오기 (배치로)
  │             fetch.min.bytes / fetch.max.wait.ms에 따라 타이밍 결정
  │
  ▼
② Deserializer ─── 바이트 배열을 객체로 변환 (key, value 각각)
  │
  ▼
③ ConsumerInterceptor ─── 메시지 가로채기 (선택사항)
  │
  ▼
④ 비즈니스 로직 처리 ─── @KafkaListener 메서드 실행
  │
  ▼
⑤ Offset Commit ─── "여기까지 읽었다"를 브로커에 기록
                      auto.commit vs 수동 commit
```

### Spring 비유

```
Spring MVC:    DispatcherServlet이 요청을 받아서 → @Controller 메서드에 전달
Spring Kafka:  KafkaMessageListenerContainer가 poll()로 메시지를 받아서 → @KafkaListener 메서드에 전달
```

차이점: HTTP는 요청이 "밀어오는(push)" 방식이고, Kafka 컨슈머는 직접 "당겨오는(pull)" 방식이다.

**Pull 방식의 장점:**
- 컨슈머가 자기 처리 속도에 맞춰서 가져감 → 과부하 방지
- 배치로 가져올 수 있음 → 처리 효율 향상

## 핵심 개념: 컨슈머 그룹

### 왜 컨슈머 그룹이 필요한가?

```
컨슈머 그룹이 없다면:
  컨슈머 A가 파티션 0,1,2를 모두 읽음
  컨슈머 B도 파티션 0,1,2를 모두 읽음
  → 같은 메시지를 중복 처리 (의도하지 않은 경우)

컨슈머 그룹이 있으면:
  그룹 "payment-service":
    컨슈머 A ← 파티션 0, 1
    컨슈머 B ← 파티션 2
    → 파티션이 나눠지므로 중복 처리 없음, 병렬 처리 가능

  그룹 "notification-service":
    컨슈머 C ← 파티션 0, 1, 2 (독립적으로 전부 읽음)
    → 다른 그룹이므로 payment-service와 무관
```

### 핵심 규칙

```
1. 하나의 파티션은 같은 그룹 내에서 하나의 컨슈머만 읽을 수 있다
   → 파티션 수 = 그룹 내 최대 병렬 처리 수

2. 파티션 수보다 컨슈머가 많으면 놀게 된다
   파티션 3개, 컨슈머 5개 → 2개는 유휴 상태

3. 컨슈머가 추가/제거되면 리밸런싱이 발생한다
   → 파티션 재분배 → 이 동안 모든 컨슈머가 일시 정지
```

### 컨슈머 수에 따른 파티션 할당 변화

```
파티션 3개 (P0, P1, P2) 기준:

컨슈머 1대:
  C1 ← P0, P1, P2  (혼자 다 처리)

컨슈머 2대:
  C1 ← P0, P1
  C2 ← P2

컨슈머 3대 (최적):
  C1 ← P0
  C2 ← P1
  C3 ← P2

컨슈머 4대:
  C1 ← P0
  C2 ← P1
  C3 ← P2
  C4 ← 없음 (유휴!)
```

## 핵심 설정 — 카테고리별 정리

### 카테고리 1: 컨슈머 그룹

| 설정 | 기본값 | 설명 |
|---|---|---|
| `group.id` | 없음 (필수) | 컨슈머 그룹 이름. 같은 group.id = 같은 그룹 |
| `group.instance.id` | null | Static Membership용. 설정하면 리밸런싱 최소화 |

### 카테고리 2: 오프셋 관리

| 설정 | 기본값 | 설명 |
|---|---|---|
| `auto.offset.reset` | `latest` | 오프셋이 없을 때 어디서부터 읽을지 |
| `enable.auto.commit` | `true` | 오프셋 자동 커밋 여부 |
| `auto.commit.interval.ms` | `5000` | 자동 커밋 간격 |

### 카테고리 3: 가져오기(Fetch) 설정

| 설정 | 기본값 | 설명 |
|---|---|---|
| `fetch.min.bytes` | `1` | 최소 이만큼 쌓여야 가져옴 |
| `fetch.max.wait.ms` | `500` | fetch.min.bytes 안 차도 이 시간 지나면 가져옴 |
| `max.poll.records` | `500` | poll() 한 번에 가져오는 최대 레코드 수 |
| `max.partition.fetch.bytes` | `1048576` (1MB) | 파티션당 한 번에 가져오는 최대 크기 |

### 카테고리 4: 세션 / 하트비트

| 설정 | 기본값 | 설명 |
|---|---|---|
| `session.timeout.ms` | `45000` | 이 시간 동안 하트비트 없으면 죽은 것으로 판단 |
| `heartbeat.interval.ms` | `3000` | 하트비트 전송 간격 |
| `max.poll.interval.ms` | `300000` (5분) | poll() 간격이 이걸 초과하면 죽은 것으로 판단 |

## 각 설정 상세 설명

### auto.offset.reset — 어디서부터 읽을 것인가

```
이 설정이 적용되는 시점:
  → 컨슈머 그룹이 처음 토픽을 구독할 때 (커밋된 오프셋이 없을 때)
  → 커밋된 오프셋이 유효하지 않을 때 (retention으로 삭제된 경우)

earliest:
  파티션의 처음(가장 오래된 메시지)부터 읽음
  → 과거 데이터도 모두 처리해야 할 때
  → 새 서비스 추가 시 기존 데이터 소비 목적

latest (기본값):
  현재 시점 이후의 새 메시지만 읽음
  → 과거 데이터가 필요 없을 때
  → 실시간 처리 목적

none:
  커밋된 오프셋이 없으면 예외 발생
  → 명시적으로 오프셋 관리를 강제하고 싶을 때
```

### enable.auto.commit — 오프셋 커밋 전략 (가장 중요한 설정)

오프셋 커밋 = "여기까지 읽었다"를 기록하는 것.
이 설정을 잘못하면 **메시지 유실 또는 중복 처리**가 발생한다.

#### 자동 커밋 (enable.auto.commit=true, 기본값)

```
poll() → 메시지 100건 가져옴 → 처리 중...
                                    ↑ 5초 지남 (auto.commit.interval.ms)
                                    ↑ 자동으로 오프셋 커밋!
                                    → 아직 50건밖에 처리 안 했는데 100건 다 읽은 걸로 기록됨

만약 이 시점에 컨슈머가 죽으면?
  → 나머지 50건은 처리 안 됐지만, 오프셋은 이미 커밋됨
  → 메시지 유실!
```

#### 수동 커밋 — Commit 시점을 직접 제어

```
방법 1: commitSync() — 동기 커밋
  처리 완료 → commitSync() → 커밋 완료 확인 → 다음 poll()
  → 가장 안전하지만, 커밋할 때마다 블로킹

방법 2: commitAsync() — 비동기 커밋
  처리 완료 → commitAsync() → 바로 다음 poll()
  → 빠르지만, 커밋 실패 시 감지가 어려움

방법 3: 실무 권장 조합
  평소에는 commitAsync() → 셧다운 시 commitSync()
```

## 오프셋 커밋 전략 심화

### 전략 1: 레코드 단위 커밋 (가장 안전, 가장 느림)

```java
@KafkaListener(topics = "order-events", groupId = "payment-service")
public void listen(ConsumerRecord<String, String> record, Acknowledgment ack) {
    try {
        processOrder(record.value());
        ack.acknowledge();  // 이 메시지 처리 완료 후 커밋
    } catch (Exception e) {
        // 커밋하지 않음 → 다음 poll()에서 같은 메시지를 다시 받음
        log.error("처리 실패, 재처리 예정: offset={}", record.offset(), e);
    }
}
```

### 전략 2: 배치 단위 커밋 (균형잡힌 선택)

```java
@KafkaListener(topics = "order-events", groupId = "payment-service")
public void listen(List<ConsumerRecord<String, String>> records, Acknowledgment ack) {
    try {
        for (ConsumerRecord<String, String> record : records) {
            processOrder(record.value());
        }
        ack.acknowledge();  // 배치 전체 처리 완료 후 한 번만 커밋
    } catch (Exception e) {
        // 배치 전체를 재처리해야 함 → 멱등성 필요
        log.error("배치 처리 실패", e);
    }
}
```

### 전략 3: 자동 커밋 (가장 단순, 유실 가능)

```java
// enable.auto.commit=true일 때
@KafkaListener(topics = "order-events", groupId = "log-service")
public void listen(String message) {
    // 별도 ack 필요 없음. 자동으로 커밋됨
    // 유실되어도 괜찮은 데이터 (로그 등)에만 사용
    saveLog(message);
}
```

## 리밸런싱 (Rebalancing)

### 리밸런싱이 발생하는 시점

```
1. 컨슈머 추가: 새 인스턴스 배포, 스케일 아웃
2. 컨슈머 제거: 인스턴스 종료, 장애
3. 토픽 변경: 파티션 수 증가
4. 컨슈머가 죽은 것으로 판단:
   - session.timeout.ms 동안 하트비트 없음
   - max.poll.interval.ms 동안 poll() 호출 없음
```

### 리밸런싱의 문제점

```
리밸런싱 중:
  모든 컨슈머가 읽기를 멈춤 (Stop The World)
  → 파티션 재분배 완료까지 메시지 소비 불가
  → K8s 롤링 배포 시 Pod 교체마다 리밸런싱 반복 가능

시간: 수 초 ~ 수십 초 (컨슈머/파티션 수에 비례)
```

### 리밸런싱 전략

#### Eager Rebalancing (기본, Kafka 2.3 이전)

```
리밸런싱 시작 → 모든 컨슈머가 파티션을 반납 → 전체 재분배
→ 모든 컨슈머가 잠시 멈춤 (Stop The World)

C1: [P0, P1] → [    ] → [P0]      ← P1을 잠시라도 잃음
C2: [P2]     → [    ] → [P1, P2]
```

#### Cooperative Sticky Rebalancing (Kafka 2.4+, 권장)

```
리밸런싱 시작 → 변경이 필요한 파티션만 이동 → 나머지는 계속 처리
→ 영향 최소화

C1: [P0, P1] → [P0]     ← P0은 계속 읽음 (중단 없음!)
C2: [P2]     → [P1, P2] ← P1만 새로 할당
```

설정 방법:
```yaml
spring:
  kafka:
    consumer:
      properties:
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor
```

### Static Group Membership — 리밸런싱 최소화

```
일반 컨슈머:
  컨슈머 재시작 → 그룹 탈퇴 → 리밸런싱 → 그룹 재가입 → 리밸런싱
  → 재시작할 때마다 리밸런싱 2번

Static Membership (group.instance.id 설정):
  컨슈머 재시작 → session.timeout.ms 내에 복귀하면 리밸런싱 없음
  → 같은 파티션을 그대로 할당받음
```

```yaml
spring:
  kafka:
    consumer:
      properties:
        group.instance.id: payment-consumer-1  # 인스턴스마다 고유한 값
        session.timeout.ms: 30000              # 이 시간 내에 복귀하면 리밸런싱 안 함
```

## session.timeout.ms vs max.poll.interval.ms

이 둘을 헷갈리면 장애 원인을 못 찾는다.

```
session.timeout.ms (기본 45초):
  → 하트비트 기반. 백그라운드 스레드가 보냄.
  → "이 컨슈머 프로세스가 살아있는가?"
  → 네트워크 장애, 프로세스 죽음 감지

max.poll.interval.ms (기본 5분):
  → poll() 호출 간격 기반.
  → "이 컨슈머가 메시지를 처리하고 있는가?"
  → 처리가 너무 오래 걸리는 경우 감지

실전 시나리오:
  컨슈머가 메시지 하나를 처리하는데 6분 걸림
  → 하트비트는 정상 (프로세스는 살아있으니까)
  → 하지만 5분 동안 poll()을 안 했으므로
  → max.poll.interval.ms 초과 → 리밸런싱 발생!
  → 처리 중이던 메시지는 다른 컨슈머에게 재할당 → 중복 처리 위험

해결:
  1. max.poll.interval.ms를 늘린다 (임시방편)
  2. max.poll.records를 줄여서 처리 시간을 단축한다 (권장)
  3. 무거운 처리는 별도 스레드풀로 위임한다 (근본적 해결)
```

## fetch 관련 설정 — 처리량 튜닝

```
fetch.min.bytes = 1 (기본값)
fetch.max.wait.ms = 500 (기본값)

기본값일 때:
  데이터 1바이트라도 있으면 즉시 가져옴
  → 지연 최소화 / 네트워크 왕복 많음

fetch.min.bytes = 50000 (50KB)로 변경하면:
  50KB 이상 쌓이거나 500ms 지나면 가져옴
  → 배치로 가져오므로 처리량 증가 / 지연 약간 증가

max.poll.records = 500 (기본값):
  poll() 한 번에 최대 500건
  → 줄이면: 처리 시간 단축 → max.poll.interval.ms 초과 방지
  → 늘리면: 처리량 증가 / 메모리 사용 증가
```

## 실무에서 반드시 챙겨야 할 것들

### 1. 오프셋 커밋 전략은 데이터 중요도에 맞춰라

```
유실 허용 (로그, 메트릭):
  → enable.auto.commit=true (기본값)
  → 가장 단순, 약간의 유실 가능

유실 불허 (결제, 주문):
  → enable.auto.commit=false + 수동 커밋
  → 처리 완료 후 커밋 → 유실 없음, 중복 가능
  → 컨슈머 로직에 멱등성 필수
```

### 2. max.poll.interval.ms와 max.poll.records 균형

```
❌ 흔한 실수:
  max.poll.records = 500 (기본값) 그대로 두고
  메시지 하나당 처리 시간이 1초
  → 500건 × 1초 = 500초 = 8분 이상
  → max.poll.interval.ms(5분) 초과 → 리밸런싱!

✅ 해결:
  방법 1: max.poll.records를 줄인다 (예: 50건)
  방법 2: 처리 로직을 비동기로 위임하고 poll()을 빠르게 돌린다
  방법 3: max.poll.interval.ms를 늘린다 (임시방편, 비권장)
```

### 3. Graceful Shutdown — 컨슈머 종료 시 오프셋 잃지 않기

```java
@Configuration
public class KafkaConsumerConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, String> kafkaListenerContainerFactory(
            ConsumerFactory<String, String> consumerFactory) {
        ConcurrentKafkaListenerContainerFactory<String, String> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);

        // Graceful shutdown: 처리 중인 메시지 완료 후 종료
        ContainerProperties props = factory.getContainerProperties();
        props.setAckMode(ContainerProperties.AckMode.MANUAL_IMMEDIATE);
        props.setShutdownTimeout(10000L); // 종료 시 최대 10초 대기

        return factory;
    }
}
```

### 4. Dead Letter Topic (DLT) — 처리 실패 메시지 관리

```
메시지 처리 실패 시:
  ❌ 무한 재시도 → 그 메시지에 걸려서 전체 소비 정지
  ❌ 무시하고 넘김 → 데이터 유실

  ✅ 재시도 N번 → 실패 시 Dead Letter Topic으로 이동 → 다음 메시지 처리 계속
```

```java
@Configuration
public class KafkaConsumerConfig {

    @Bean
    public DefaultErrorHandler errorHandler(KafkaTemplate<String, String> kafkaTemplate) {
        // 실패한 메시지를 DLT(Dead Letter Topic)로 보냄
        DeadLetterPublishingRecoverer recoverer =
            new DeadLetterPublishingRecoverer(kafkaTemplate);

        // 3번 재시도 후 DLT로 이동
        DefaultErrorHandler handler = new DefaultErrorHandler(
            recoverer,
            new FixedBackOff(1000L, 3)  // 1초 간격으로 3번 재시도
        );

        // 역직렬화 에러는 재시도 의미 없음 → 바로 DLT로
        handler.addNotRetryableExceptions(DeserializationException.class);

        return handler;
    }
}
```

```
토픽 구조:
  order-events          ← 원본 토픽
  order-events.DLT      ← Dead Letter Topic (Spring Kafka가 자동 생성)

실패한 메시지는 DLT에 쌓이고, 나중에 원인 분석 후 재처리하면 됨
```

### 5. 컨슈머 Lag 모니터링 — 가장 중요한 메트릭

```
Lag = 프로듀서가 쓴 최신 offset - 컨슈머가 읽은 offset

Lag = 0:  컨슈머가 실시간으로 따라가고 있음 (이상적)
Lag = 1000: 1000건 밀려 있음
Lag 증가 추세: 컨슈머 처리 속도 < 프로듀서 생산 속도 → 조치 필요!
```

확인 방법:
```bash
docker exec kafka-1 kafka-consumer-groups \
  --bootstrap-server kafka-1:29092 \
  --group payment-service \
  --describe
```

출력 예시:
```
GROUP            TOPIC         PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
payment-service  order-events  0          45              50              5
payment-service  order-events  1          30              30              0
payment-service  order-events  2          28              35              7
```

**Lag가 계속 늘어나면:**
1. 컨슈머 인스턴스 추가 (파티션 수 이내에서)
2. max.poll.records 조정
3. 처리 로직 최적화
4. 파티션 수 자체를 늘리기 (마지막 수단, key 라우팅이 깨질 수 있음)

### 6. 실무 권장 설정 조합

```yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092,localhost:9093,localhost:9094
    consumer:
      group-id: payment-service
      auto-offset-reset: earliest
      enable-auto-commit: false              # 수동 커밋 (데이터 중요할 때)
      max-poll-records: 100                  # poll() 당 최대 100건
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      properties:
        # 리밸런싱 최적화
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor
        # 세션 / 하트비트
        session.timeout.ms: 45000
        heartbeat.interval.ms: 3000
        max.poll.interval.ms: 300000
        # Fetch 튜닝
        fetch.min.bytes: 1
        fetch.max.wait.ms: 500
    listener:
      ack-mode: manual-immediate             # 수동 커밋 모드
      concurrency: 3                         # 컨슈머 스레드 수 (파티션 수에 맞춤)
```

## Spring Kafka 컨슈머 실습 코드

### 1. 기본 컨슈머 (자동 커밋)

```java
@Component
public class SimpleConsumer {

    /**
     * 가장 단순한 컨슈머
     * enable.auto.commit=true일 때 사용
     * 유실 허용 가능한 데이터에 적합
     */
    @KafkaListener(topics = "test-topic", groupId = "simple-group")
    public void listen(String message) {
        log.info("수신: {}", message);
    }
}
```

### 2. 수동 커밋 컨슈머 (실무 권장)

```java
@Component
public class ManualCommitConsumer {

    /**
     * 수동 커밋 컨슈머
     * 처리 완료 후 명시적으로 ack → 유실 방지
     * enable.auto.commit=false + ack-mode=manual-immediate 필요
     */
    @KafkaListener(topics = "order-events", groupId = "payment-service")
    public void listen(ConsumerRecord<String, String> record, Acknowledgment ack) {
        try {
            log.info("수신: partition={}, offset={}, key={}, value={}",
                record.partition(), record.offset(), record.key(), record.value());

            processPayment(record.value());

            ack.acknowledge();  // 처리 완료 후 커밋
        } catch (Exception e) {
            log.error("처리 실패: offset={}", record.offset(), e);
            // ack 안 하면 → 다음 poll()에서 재수신 (또는 에러핸들러가 처리)
        }
    }
}
```

### 3. 배치 컨슈머

```java
@Component
public class BatchConsumer {

    /**
     * 배치로 메시지를 받아서 한 번에 처리
     * spring.kafka.listener.type=batch 설정 필요
     * DB bulk insert 등에 유용
     */
    @KafkaListener(topics = "log-events", groupId = "log-service")
    public void listen(List<ConsumerRecord<String, String>> records, Acknowledgment ack) {
        log.info("배치 수신: {}건", records.size());

        List<LogEntity> logs = records.stream()
            .map(record -> parseLog(record.value()))
            .toList();

        logRepository.saveAll(logs);  // bulk insert

        ack.acknowledge();  // 배치 전체 커밋
    }
}
```

### 4. 특정 파티션 지정 소비

```java
@Component
public class PartitionAssignedConsumer {

    /**
     * 특정 파티션만 직접 소비
     * 컨슈머 그룹을 사용하지 않으므로 리밸런싱 없음
     * 특수한 경우에만 사용 (특정 파티션의 데이터만 필요할 때)
     */
    @KafkaListener(
        topicPartitions = @TopicPartition(
            topic = "order-events",
            partitions = {"0", "1"}  // 파티션 0, 1만 읽음
        ),
        groupId = "partition-specific-group"
    )
    public void listen(ConsumerRecord<String, String> record) {
        log.info("파티션 {} 전용 소비: offset={}", record.partition(), record.offset());
    }
}
```

### 5. 컨슈머 Lag 확인용 코드

```java
@Component
@RequiredArgsConstructor
public class ConsumerLagChecker {

    private final ConsumerFactory<String, String> consumerFactory;

    /**
     * 프로그래밍적으로 컨슈머 Lag 확인
     * 모니터링 API 엔드포인트로 노출할 때 유용
     */
    public Map<TopicPartition, Long> checkLag(String groupId, String topic) {
        try (Consumer<String, String> consumer = consumerFactory.createConsumer(groupId, "lag-checker")) {
            List<TopicPartition> partitions = consumer.partitionsFor(topic).stream()
                .map(info -> new TopicPartition(topic, info.partition()))
                .toList();

            Map<TopicPartition, Long> endOffsets = consumer.endOffsets(partitions);
            // committed offsets vs end offsets 비교하여 lag 계산
            return endOffsets;
        }
    }
}
```

## 컨슈머 관련 장애 시나리오

### 시나리오 1: 무한 리밸런싱

```
증상: 컨슈머가 계속 리밸런싱을 반복하며 메시지를 못 읽음
원인: 메시지 처리 시간 > max.poll.interval.ms
  → 리밸런싱 → 같은 메시지 재할당 → 또 시간 초과 → 또 리밸런싱...

해결:
  1. max.poll.records를 줄인다 (예: 500 → 50)
  2. 무거운 처리는 별도 스레드로 위임
  3. max.poll.interval.ms를 늘린다 (임시)
```

### 시나리오 2: 중복 처리

```
증상: 같은 메시지가 2번 처리됨
원인:
  - 처리 완료 후 커밋 전에 리밸런싱 발생
  - 다른 컨슈머가 같은 메시지를 다시 받아 처리

해결:
  - 컨슈머 로직에 멱등성 구현 (DB unique constraint, 중복 체크 등)
  - Kafka는 "at least once"가 기본이므로 중복은 항상 가능하다고 가정
```

### 시나리오 3: 하나의 파티션이 처리 안 됨

```
증상: 특정 파티션의 Lag만 계속 증가
원인: 해당 파티션의 메시지가 역직렬화 실패 또는 처리 예외
  → 에러 핸들러가 없으면 같은 메시지에서 계속 멈춤 (Poison Pill)

해결:
  - DefaultErrorHandler + Dead Letter Topic 설정
  - 실패 메시지를 DLT로 보내고 다음 메시지 처리 계속
```

### 시나리오 4: K8s 롤링 배포 시 리밸런싱 폭풍

```
증상: 배포할 때마다 수십 초간 메시지 소비 중단
원인:
  Pod 1 종료 → 리밸런싱 → Pod 2 종료 → 리밸런싱 → ...
  Eager 리밸런싱이면 매번 전체 정지

해결:
  1. CooperativeStickyAssignor 사용 → 영향받는 파티션만 이동
  2. Static Group Membership → session.timeout.ms 내 복귀 시 리밸런싱 없음
  3. Pod의 graceful shutdown 설정 → 오프셋 커밋 후 종료
```
